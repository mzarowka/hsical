# hsical v2.2 — decouple defaults from app.R

**Date:** 2026-07-15
**Type:** design proposal (not started)
**Against:** v2.1 (`inst/app/app.R`)

## Motivation

`app.R` hardcodes deployment/site-specific values in source. Clearest offender: the
lens list still ships two literal placeholders — `"VNIR lens 1 (TODO)"`,
`"VNIR lens 2 (TODO)"` (`app.R:447`). A lab adding its own objectives has to edit
source. `manufacturer = "Specim"` is hardcoded *and duplicated* (UI `:438`,
clear-session `:999`). Sensor choices `c("VNIR","SWIR")` (`:430`) and theme primary
`#2c6e8f` (`:278`) are the same kind of thing.

Goal: make app source identical across labs; move site-specific knowledge into one
editable file a non-R user can maintain.

## Scope discipline (important)

Two piles of "defaults", only one moves:

- **Config (externalize):** choice-lists + scalar defaults — lens list, manufacturer,
  sensor-type choices, maybe theme colour.
- **Structural (stays in code):** five-numbers-in/four-out geometry, field lists
  (`SESSION_*`, `PER_CAPTURE_*`, `REVIEW_*`), validation. Moving these would break the
  "hsical owns no schema" principle.

Config supplies **choice lists + simple scalar defaults only** — not field
definitions, not layout, not validation. That boundary keeps v2.2 from drifting into a
templating engine.

## Two features hiding in "read yaml with your lenses"

1. **Defaults/config file** — dedicated `defaults.yaml` read once at startup to
   populate lens list / manufacturer / sensor choices. *App configuration.*
2. **Template sidecar** — pick an existing HSItools *sidecar* (same YAMLs the Review
   panel reads) and adopt its session-stable fields as prefills. *Data reuse.*

Look alike, aren't. Keep separate — the app already reads sidecar YAML in Review, and
conflating "config" with "an HSItools schema object" muddies exactly the boundary the
package is proud of.

**Recommendation: build #1 for v2.2.** It directly kills the TODO placeholders and the
duplicated "Specim". Treat #2 as an optional later convenience.

> Caveat under ecosystem §1: HSItools is schema/serialization owner. A hsical
> `defaults.yaml` is *app config*, not sidecar data — it must never grow its own copy
> of the sidecar field list. If the config file starts to look like a schema, stop.

## Concrete shape for #1

- Ship packaged `inst/app/defaults.yaml` with real values (finally fill in VNIR lens
  names).
- Discovery precedence, missing-file-safe:
  `run_app(defaults=)` arg → `HSICAL_DEFAULTS` env var →
  `tools::R_user_dir("hsical","config")/defaults.yaml` → packaged default.
- Missing/malformed user file → fall back to packaged defaults with a
  `showNotification`, never crash. A lab with zero config still gets a working app.
- Flat structure first:
  ```yaml
  manufacturer: Specim
  sensor_types: [VNIR, SWIR]
  lenses: [OLES30, OLESmacro, ...]   # real VNIR names here
  theme_primary: "#2c6e8f"
  ```
  Named "instrument profiles" only if more than the one Specim rig is supported —
  otherwise premature.

## Touch points when implemented

- `selectizeInput("lens", choices=...)` `app.R:442-455`
- `selectizeInput("sensor_type", choices=...)` `app.R:427-436`
- `textInput("manufacturer", value=...)` `app.R:437-440` **and** clear-session
  `updateTextInput(... "manufacturer", value="Specim")` `app.R:999` — single source of
  truth removes this duplication.
- `bs_theme(primary=...)` `app.R:278`
- `run_app()` gains a `defaults=` passthrough `R/run_app.R:14`

## Open questions

- Which YAML reader? hsical currently has no `yaml::` dependency by design (schema
  machinery is HSItools'). Reading *config* is not reading *sidecars*, so a direct
  `yaml` import is defensible — but confirm with Maury it doesn't blur the boundary,
  and check whether reusing an HSItools-exported reader is cleaner than a new dep.
- Should theme colour be user-configurable, or scope-creep past the instrument metadata
  that actually varies? Lean: include it, cheap.
- Per ecosystem §0: this is design-first / ask-before-locking territory. Nothing here
  is settled — it's a proposal to discuss, not a plan to execute.
